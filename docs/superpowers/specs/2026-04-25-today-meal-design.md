# Today Meal 精致私人菜单小程序设计

## 背景与目标

Today Meal 是一个给两个人使用的微信小程序：它既是一本漂亮的私人菜单册，也能在“不知道今天吃什么”时给出推荐。

第一版聚焦两个问题：

- 记录自家会做的菜、食谱和图片，让她可以挑想吃什么。
- 维护常点或想点的外卖，支持链接跳转或复制，并通过随机推荐降低选择成本。

产品气质选择“清新杂志感”：浅色、留白、柔和卡片、漂亮食物图、清晰分类。整体要像一本为她准备的菜单杂志，而不是后台管理工具。

## 第一版范围

第一版采用“精致私人菜单 MVP”路线，包含：

- 今日推荐首页：顶部展示推荐卡片，下方浏览菜单。
- 菜单库：支持自家菜谱、外卖店菜、灵感清单三类内容。
- 图片与标签：每个菜单项可上传图片，可绑定餐段和自定义标签。
- 轻智能抽签：根据标签、餐段、收藏、评分、最近吃过记录生成随机推荐。
- 吃过历史：记录吃过时间、喜欢程度和备注，用于后续推荐。
- 外卖链接辅助：粘贴美团、闪购等链接后尝试提取标题和图片，失败时可手动补全。
- 权限：公开可浏览，只有已授权的两个人可以新增、编辑、标记吃过和评分。

第一版不包含：

- 多人社交、评论、关注、分享社区。
- 完整 AI 菜谱生成。
- 对美团、闪购链接的稳定全自动解析承诺。
- 营养分析、预算、购物清单、周计划。
- 单独 Web 管理后台。

## 技术架构

项目采用 monorepo：

- `apps/miniapp`：原生微信小程序 + TypeScript。
- `apps/api`：NestJS 后端服务。
- `packages/shared`：共享类型、枚举、接口契约。
- `docs`：设计文档、接口说明、部署说明。

选择原生微信小程序，是因为第一版只面向微信端，原生方案稳定、微信能力直接、抽象少，适合 Cursor/AI 后续持续维护。

后端选择 NestJS + PostgreSQL。TypeScript 前后端一致，模块边界清楚，DTO、校验、接口类型都更利于长期维护。

图片存储第一版通过后端文件服务统一抽象。部署初期可使用服务器本地挂载目录，后续可平滑迁移到 S3 兼容对象存储或其他云对象存储。

## 用户与权限

访问分为浏览和编辑两类：

- 未登录或普通访问者：可以浏览公开菜单、筛选、查看详情。
- 已授权编辑者：可以新增、编辑、上传图片、收藏、标记吃过、评分、维护标签。

登录流程使用微信小程序登录：

1. 小程序调用微信登录获取临时 code。
2. 后端用 code 换取 openid。
3. 后端判断 openid 是否已绑定编辑权限。
4. 后端返回会话 token 和用户权限。

编辑权限第一版使用“微信登录 + 邀请码绑定”。用户登录后输入预设邀请码，后端将该 openid 绑定为 editor。owner 可通过服务器环境变量预置 openid。接口层必须统一校验写操作权限。

## 信息架构与页面

底部导航建议 3 个 Tab：

- `今天`：今日推荐、抽签、精选菜单。
- `菜单`：菜单库、筛选、搜索、分类浏览。
- `我的`：登录状态、编辑权限、标签管理、最近吃过和轻量统计。

### 首页

首页采用“推荐 + 菜单混合”结构。

顶部是“今天为你选”卡片，展示推荐菜单项、封面图、标签和推荐理由。用户可以点击“换一个”或“抽一下”触发轻量动画，例如卡片洗牌、骰子或转盘感。

下方是标签筛选和菜单卡片流，保留清新杂志感的浏览体验。

### 菜单列表

菜单列表支持：

- 按类型筛选：自家菜谱、外卖店菜、灵感清单。
- 按餐段筛选：早餐、午餐、晚餐、夜宵等。
- 按标签筛选：系统标签和自定义标签。
- 按状态筛选：收藏、最近没吃、最近吃过。
- 轻量搜索：按名称、店名、备注查找。

列表视觉上使用漂亮卡片，优先展示封面图、名称、标签和关键描述。

### 菜单详情

详情页根据菜单类型展示不同字段：

- 自家菜谱：食材、步骤、预计耗时、难度、备注。
- 外卖店菜：店名、平台、价格区间、配送备注、外卖链接。
- 灵感清单：来源、备注、图片、标签。

所有详情页保留关键动作：

- 收藏或取消收藏。
- 标记今天吃了。
- 记录喜欢程度和备注。
- 编辑菜单项。

外卖链接按钮优先尝试跳转到对应小程序或可用目标；如果受平台限制，则复制链接并提示去对应 App 打开。

### 新增与编辑

新增/编辑流程按类型切换：

- 自家菜谱：图片优先，先上传或拍照，再填写名称、标签、食材、步骤。
- 外卖店菜：链接优先，先粘贴外卖链接，后端尝试生成草稿，失败时手动补店名、菜名、图片。
- 灵感清单：轻量录入，适合快速保存以后想试的菜或餐厅。

表单需要支持上传失败重试，并保留用户已填写内容。

## 后端模块与 API

后端按模块组织：

- `auth`：微信登录、openid 识别、token 签发、权限判断。
- `users`：用户资料和角色。
- `menu-items`：菜单 CRUD、筛选、搜索、收藏。
- `tags`：系统标签、自定义标签、标签绑定。
- `meal-history`：标记吃过、评分、备注、最近吃过查询。
- `recommendations`：生成今日推荐和候选列表。
- `files`：图片上传、删除、访问 URL。
- `link-preview`：外卖链接解析辅助。

关键接口包括：

- `POST /auth/wechat-login`
- `GET /menu-items`
- `POST /menu-items`
- `GET /menu-items/:id`
- `PATCH /menu-items/:id`
- `DELETE /menu-items/:id`
- `POST /menu-items/:id/favorite`
- `GET /tags`
- `POST /tags`
- `POST /meal-history`
- `GET /meal-history/recent`
- `GET /recommendations/today`
- `POST /recommendations/random`
- `POST /files/upload`
- `POST /link-preview`

接口响应使用统一格式，并用 DTO 做输入校验。写操作必须经过权限守卫。

## 推荐规则

第一版推荐不使用复杂 AI，采用可解释的轻智能规则：

1. 根据当前时间推断餐段，也允许用户手动选择早餐、午餐、晚餐、夜宵。
2. 按餐段、类型和标签筛选候选菜单。
3. 默认降低最近 N 天吃过内容的权重，或直接排除。
4. 收藏和高评分内容提高权重。
5. 在权重池中随机选出一个或多个候选。
6. 返回推荐理由，例如“最近没吃过暖胃类”“她上次给了高分”“晚餐标签匹配”。

抽签交互可以有仪式感，但规则保持简单可维护。

## 外卖链接处理

外卖链接解析作为辅助能力，不作为强依赖。

流程：

1. 用户粘贴美团、闪购或其他外卖链接。
2. 小程序调用 `POST /link-preview`。
3. 后端尝试获取标题、图片、站点和描述。
4. 成功则返回可编辑草稿。
5. 失败则返回原链接和失败原因，小程序提示用户手动补全。

由于第三方平台限制、登录态、反爬或分享链接变化，第一版不承诺自动解析所有链接，也不承诺直接完成下单。

## 错误处理

用户可见错误需要轻量友好：

- 链接解析失败：提示“没自动识别出来，可以手动补一下”，保留原链接。
- 图片上传失败：允许重试，不清空已填表单。
- 未登录编辑：允许继续浏览，点击新增或编辑时引导登录。
- 无编辑权限：提示“这个小菜单暂时只允许主人编辑”。
- 服务异常：显示轻提示，后端记录日志。

后端需要记录关键错误日志，尤其是文件上传、链接解析、推荐接口和权限失败。

## 测试策略

后端测试重点：

- 菜单 CRUD。
- 权限校验。
- 推荐规则。
- 吃过历史影响推荐。
- 链接解析失败降级。
- 文件上传接口。

小程序测试重点：

- 未登录可浏览。
- 无权限不可编辑。
- 新增三类菜单项。
- 图片上传失败重试。
- 外卖链接解析失败后的手动补全。
- 随机推荐和标记吃过流程。

第一版至少需要后端单元测试和 API e2e 测试。小程序侧配合核心逻辑测试和手动验收清单。

## 部署方案

后端部署到自购云服务器，建议使用 Docker Compose：

- `api`：NestJS 服务。
- `postgres`：PostgreSQL 数据库。
- `reverse-proxy`：Nginx 或 Caddy。
- `uploads`：图片文件挂载目录。

数据库迁移建议使用 Prisma。Prisma schema 清楚、类型生成稳定，适合 AI 后续维护。

部署文档需要覆盖：

- 环境变量。
- 数据库初始化。
- 迁移执行。
- 文件存储目录。
- HTTPS 和反向代理。
- 小程序后端域名配置。

## 成功标准

第一版完成后，应满足：

- 她打开小程序觉得漂亮、轻盈、愿意浏览。
- 你们可以稳定维护自家菜、外卖和灵感清单。
- 每天不知道吃什么时，可以通过“抽一下”得到合理建议。
- 最近吃过和评分会影响推荐，减少重复和踩雷。
- 外卖链接可以被保存，并在不能直接跳转时有清晰降级体验。
- 代码结构清楚，方便 Cursor/AI 后续继续开发。
